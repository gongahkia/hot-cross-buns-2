# MCP Agent Access Spec

## Scope

Hot Cross Buns 2 exposes an opt-in local MCP endpoint for agent clients configured by the user. The endpoint must bind to `127.0.0.1` and must never expose Google tokens, Keychain material, raw credentials, cache encryption keys, or unsanitized diagnostics.

## Endpoint

Default endpoint:

```text
http://127.0.0.1:<port>/mcp
```

Requests must include:

```text
Authorization: Bearer <generated-token>
```

The bearer token must be generated by the app and stored in the OS keychain. The user can reset it from Settings.

## Permission Modes

Required modes:

- Read-only: clients may search and read tasks, notes, events, lists, calendars, today, and week views.
- Confirm writes: clients must first dry-run a write, then pass the returned `confirmationId` to apply it.
- Allow writes: non-destructive writes can apply directly; destructive writes still require dry-run confirmation.

Every write tool must accept `dryRun`.

## Required Read Tools

- `hcb_search`
- `hcb_today`
- `hcb_week`
- `hcb_get_task`
- `hcb_get_event`
- `hcb_get_note`
- `hcb_list_task_lists`
- `hcb_list_calendars`

## Required Write Tools

- `hcb_create_task`
- `hcb_create_note`
- `hcb_create_event`
- `hcb_update_task`
- `hcb_update_note`
- `hcb_update_event`
- `hcb_complete_task`
- `hcb_reopen_task`
- `hcb_move_task`
- `hcb_delete_task`
- `hcb_delete_note`
- `hcb_delete_event`

## Response Shape

Tool responses must include:

- `applied`
- `dryRun`
- `requiresConfirmation`
- optional `confirmationId`
- `message`
- optional sanitized `item` or `items`
- optional `hotcrossbuns://` deep link

Errors must not include full request bodies, token values, raw Google responses, local file paths containing secrets, or stack traces in normal mode.

## Write Confirmation

Confirmation ids must be:

- single-use
- time-limited
- bound to the planned write payload
- bound to the requesting permission context
- invalidated after token reset or MCP server restart

Applying a confirmation must revalidate current app state before changing local data or Google-backed data.

## Security Requirements

The MCP server must:

- bind only to `127.0.0.1`
- reject unexpected browser origins
- require bearer auth for all MCP calls
- compare bearer tokens without early-exit string comparison
- cap headers and body size
- reject malformed JSON and trailing request bytes
- rate-limit requests per local client connection
- record sanitized write attempts in a local audit log
- keep audit metadata free of task titles, event details, notes, full arguments, and tokens

## Service Routing

MCP tools must call the same domain services as the UI. Do not create a separate MCP-only mutation path.

Current implementation note: MCP tool handlers are wired to the same main-side domain services used by UI IPC handlers. Task and event MCP writes therefore use the same local mirror and pending-mutation queue behavior as visible UI writes, while notes remain local-only. The user-facing MCP enable/disable IPC status is stateful, but a live listener still remains deferred until Keychain-backed bearer token storage and startup-safe server lifecycle wiring are complete.

## Tests

Required tests:

- unauthorized token rejected
- missing token rejected
- malformed JSON rejected
- oversized body rejected
- unexpected origin rejected
- read tool succeeds in read-only mode
- write tool dry-run succeeds in confirm-writes mode
- direct write is rejected in confirm-writes mode
- destructive write requires confirmation even in allow-writes mode
- audit log excludes sensitive fields
